# 用户变教练设计文档

## 一、需求说明

用户注册后默认角色为 `user`，不可自选角色。若用户希望成为教练，需提交申请，由超管（`admin`）审批。审批通过后，系统将该用户的 `role` 改为 `coach`，权限随之生效。

---

## 二、核心原则

| 原则 | 说明 |
|------|------|
| 申请表管流程 | 申请记录表负责记录申请状态（待审/通过/拒绝） |
| 库里的 role 管权限 | `users.role` 字段是权限的唯一来源 |
| JWT 与库中角色一致 | `get_current_user` 以数据库中的 `role` 为准，无需重新登录 |

---

## 三、数据表设计

### 新增表：`coach_applications`（教练申请记录）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT PK AUTO_INCREMENT | 主键 |
| `user_id` | INT FK → users.id | 申请人 ID |
| `status` | ENUM('pending','approved','rejected') | 申请状态，默认 `pending` |
| `reason` | VARCHAR(500) | 申请说明（用户填写） |
| `reject_reason` | VARCHAR(500) NULL | 拒绝原因（超管填写） |
| `reviewed_by` | INT FK → users.id NULL | 审核人 ID |
| `created_at` | DATETIME | 申请时间 |
| `updated_at` | DATETIME | 最后更新时间 |

**约束：**
- `user_id` + `status='pending'` 唯一：一人只能有一条待审申请（通过唯一索引或业务层校验）
- 已是 `coach` 的用户不可再申请（接口层校验）

---

## 四、接口设计

### 用户端

#### 4.1 提交申请
```
POST /api/coach-application/apply
权限：user（已登录，role=user）
Body：{ "reason": "申请说明" }
校验：
  - 当前 role 已是 coach → 返回错误
  - 已有 pending 申请 → 返回错误
返回：{ code: 200, msg: "申请已提交", data: { id, status, created_at } }
```

#### 4.2 查询自己的申请状态
```
GET /api/coach-application/my
权限：已登录用户
返回：{ code: 200, msg: "ok", data: { id, status, reason, reject_reason, created_at, updated_at } }
       无申请时 data: null
```

---

### 管理端（超管）

#### 4.3 获取申请列表
```
GET /api/admin/coach-applications?status=pending&page=1&size=20
权限：admin
Query：status 可选（pending/approved/rejected），不传则返回全部
返回：{ code: 200, msg: "ok", data: { total, list: [...] } }
```

#### 4.4 通过申请
```
POST /api/admin/coach-applications/{application_id}/approve
权限：admin
操作：
  1. 将 application.status → approved，reviewed_by = 当前管理员 id，updated_at = now
  2. 将 users.role → coach
返回：{ code: 200, msg: "已通过", data: null }
```

#### 4.5 拒绝申请
```
POST /api/admin/coach-applications/{application_id}/reject
权限：admin
Body：{ "reject_reason": "拒绝原因" }
操作：
  1. 将 application.status → rejected，reject_reason 写入，reviewed_by = 当前管理员 id
  2. users.role 不变
返回：{ code: 200, msg: "已拒绝", data: null }
```

## 五、鉴权方案

**采用「数据库 role 为准」方案，无需重新登录。**

```
get_current_user(token) 流程：
  1. 解析 JWT → 取 user_id
  2. 查数据库 users.role（不信任 token 中的 role 字段）
  3. 返回带最新 role 的用户对象
```

**优点：**
- 审批通过后立即生效，用户无需重新登录
- token 不需要包含 role，减少 token 失效问题

**注意：**
- 若 JWT 中存了 role，前端路由跳转依赖 token 中的 role 时，需在审批通过后提示用户刷新或重新登录以更新前端状态
- 推荐：前端登录后将 role 存入 Vuex，审批通过后调用「刷新用户信息」接口更新 Vuex 中的 role，无需重登

## 六、前端页面设计

### 用户端

#### 6.1 申请成为教练页面（`/apply-coach`）
- 展示当前申请状态（无申请 / 待审中 / 已通过 / 已拒绝+原因）
- 无申请或已拒绝时：显示申请表单（填写申请说明 + 提交按钮）
- 待审中：显示"审核中，请耐心等待"
- 已通过：显示"您已是教练，前往教练端"

#### 6.2 路由控制
- 教练模块路由守卫：检查 Vuex 中 `user.role === 'coach'`
- 申请状态可在导航栏或个人中心入口处展示提示

---

### 管理端

#### 6.3 教练申请管理页面（`/admin/coach-applications`）
- Tab 切换：待审 / 已通过 / 已拒绝
- 列表字段：申请人、申请时间、申请说明、操作（通过/拒绝）
- 拒绝时弹出 Dialog 填写拒绝原因

---

## 七、业务流程图

```
用户提交申请
    ↓
coach_applications 新增一条 pending 记录
    ↓
超管在管理端看到待审列表
    ↓
    ├── 点击「通过」→ application.status=approved + users.role=coach
    │                → 用户下次请求时 get_current_user 返回 role=coach
    │
    └── 点击「拒绝」→ application.status=rejected + reject_reason 写入
                     → users.role 不变，用户可重新申请
```

---

## 八、边界情况处理

| 场景 | 处理方式 |
|------|----------|
| 用户已是 coach，再次申请 | 接口返回错误：您已是教练 |
| 用户有 pending 申请，再次提交 | 接口返回错误：您已有待审申请 |
| 被拒绝后重新申请 | 允许，创建新的 pending 记录 |
| 超管对已处理的申请再次操作 | 接口校验 status，非 pending 则返回错误 |
| 审批通过后用户 token 中 role 仍是 user | get_current_user 以数据库为准，权限正常；前端需刷新用户信息 |
