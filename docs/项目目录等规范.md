# FastAPI 后端项目目录与接口规范

## 一、目标

将 FastAPI 项目按分层与目录约定组织，使 **ORM、校验、配置、工具、数据库访问、路由** 职责清晰，且 **所有 HTTP 接口返回统一信封结构**，便于前端与自动化处理。

---

## 二、目录结构约定

项目根目录下建议采用如下结构（可按业务增减子模块，但职责不得混淆）：

```
项目根/
├── main.py                 # 应用入口：创建 App、中间件、挂载路由、生命周期
├── config/                 # 配置（环境变量、Settings、常量）
│   └── settings.py         # 例：pydantic-settings / 自定义配置类
├── models/                 # 仅放数据库 ORM 模型（SQLAlchemy 等），一表一类或多表分文件
│   ├── __init__.py         # 统一导出，供 Base.metadata / Alembic 发现模型
│   ├── user.py             # 例：User 表
│   └── ...
├── schemas/                # 仅放 Pydantic（请求体、响应体、校验模型），禁止写 SQL/CRUD
│   ├── user.py
│   └── ...
├── CRUD/                   # 仅放对「单表/聚合」的增删改查封装，接收 Session，返回 ORM 或标量
│   ├── __init__.py
│   ├── user.py             # 例：get_user_by_id, create_user, ...
│   └── ...
├── api/ 或 routers/        # 路由：参数解析、依赖注入、调用 CRUD、组装统一响应
│   ├── user.py
│   └── ...
├── db/                     # 引擎、Session、Base（若已存在可保留）
│   └── session.py
└── utils/                  # 与业务弱耦合的纯工具（JWT、日期、字符串、统一响应封装等）
    └── ...
```

### 各目录职责（必须遵守）

| 目录 | 允许 | 禁止 |
|------|------|------|
| **models/** | ORM 类、`__tablename__`、关系、索引声明 | Pydantic 模型、业务逻辑、直接写 HTTP 响应 |
| **schemas/** | `BaseModel`、Field 校验、枚举、嵌套结构 | ORM 类、数据库 Session 操作 |
| **CRUD/** | 通过 Session 对表做增删改查、简单查询组合 | FastAPI `Request/Response`、路由装饰器 |
| **config/** | `Settings`、`.env` 映射、环境区分、常量配置 | 业务 SQL、具体接口逻辑 |
| **utils/** | 可复用函数、与表无关的辅助逻辑 | 单表专属 CRUD（应放 CRUD） |
| **api/** | 路由、依赖、调用 CRUD/schemas、返回统一格式 | 在路由里写长 SQL（应下沉到 CRUD） |

**说明：** 若当前配置在 `core/config.py` 等位置，重构时应 **迁移到 `config/`** 并更新引用，或保留 `core` 仅作兼容层再逐步废弃，以符合「配置在 config 目录」的约定。

---

## 三、分层调用关系

```
HTTP 请求
  → api/ 路由（解析 schemas、鉴权依赖）
  → CRUD/（使用 db Session 访问 models）
  → models/（ORM）
  ← 路由层将结果装入统一响应（utils 或专用 response 模块）
```

- 路由层：**不**直接操作 `session.execute` 等底层 SQL（除非极简单的健康检查且团队允许）。
- CRUD 层：函数签名建议显式传入 `Session` 或 `AsyncSession`，便于测试与复用。

---

## 四、统一接口返回格式（强制）

所有业务接口（含列表、分页、空数据）的 JSON 根对象 **必须为**：

```json
{
  "code": 200,
  "message": "success",
  "data": null
}
```

### 字段约定

- **code**（整数）：业务/HTTP 语义状态码。建议与团队约定一套表，例如：`200` 成功、`4xx` 客户端问题、`5xx` 服务端问题；也可与 HTTP status 对齐或单独定义业务码，但 **全项目一致**。
- **message**（字符串）：人类可读说明；成功可为固定文案如 `"success"`，失败为具体原因（可对用户展示或脱敏后展示）。
- **data**（任意 JSON 可序列化类型）：成功时的载荷；无数据时用 `null`、空对象 `{}` 或空数组 `[]`，**由项目统一约定一种**，避免同一含义多种写法。

### 错误响应

失败时仍使用同一信封，例如：

```json
{
  "code": 400,
  "message": "参数校验失败：邮箱格式不正确",
  "data": null
}
```

路由层应通过 **统一封装函数**（建议放在 `utils/response.py` 或同类模块）生成字典或 `JSONResponse`，避免每个接口手写字典键名不一致。

**注意：** 若历史代码使用 `msg` 等字段名，应按本规范统一为 **`message`**，并同步前端约定。

---

## 五、命名与文件组织建议

- **models**：文件名与业务域一致，如 `user.py` 对应用户相关表；多表可同文件或拆分子模块 `models/user/`。
- **schemas**：与 API 入参/出参对应，可后缀区分，如 `UserCreate`、`UserOut`、`UserListOut`。
- **CRUD**：`CRUD/user.py` 内函数名动词开头：`get_*`、`create_*`、`update_*`、`delete_*`、`list_*`。
- **路由**：REST 资源路径清晰，API 前缀（如 `/ZL-API`）在 `main.py` 或子路由统一配置。

---

## 六、重构执行清单

1. 新建 `config/`，迁移原配置模块，修正所有 `import`。
2. 确认所有 ORM 均在 `models/`，并在启动或 Alembic 中保证模型被加载。
3. 将所有 Pydantic 模型集中在 `schemas/`，路由仅引用 schemas。
4. 新建 `CRUD/`，把原写在 `api/` 或 `utils/` 里的数据库读写迁入 CRUD。
5. 统一响应：实现 `success(data=..., message=...)` / `fail(message=..., code=...)`，字段名为 **`message`**，全站路由使用。
6. 全局搜索旧字段名（如 `msg`）与散落 SQL，逐项替换。
7. 跑通启动与主要接口，与前端联调确认 `code` / `message` / `data` 解析。

---

## 七、给 AI 的一行版提示（可粘贴）

> 请按 FastAPI 分层重构本项目：`models/` 仅 ORM，`schemas/` 仅 Pydantic，`config/` 放配置，`utils/` 放通用工具，`CRUD/` 封装所有表增删改查；路由只负责入参校验、鉴权、调用 CRUD 并返回统一 JSON：`{"code": int, "message": str, "data": ...}`，禁止混用其他字段名（如 msg）。
