# 教练 · 学员 · 训练 / 饮食计划分配方案

## 一、目标（与现有诉求对齐）

1. **学员端**：教练专区——查看可绑定的教练、发起绑定申请、查看**当前绑定教练**及由其指派的**训练计划**；饮食侧可扩展为「教练建议 / 模板饮食周计划」（若饮食模块暂无「计划」实体，见下文「可选演进」）。
2. **教练端**：学员专区——学员列表与详情、处理绑定申请、为学员制定 / 维护**训练计划**，并查看学员**训练记录摘要**（可选：饮食记录摘要）。
3. **管理员端**：监管看板——教练 / 学员关系、计划与关键行为数据的可视化与审计（纠纷、合规）。

---

## 二、核心概念与数据关系（建议统一建模）

### 2.1 关系：教练–学员绑定

- 复用或强化已有 **`coach_students`**（`coach_id`, `student_id`, `status`, `bind_at`）。
- **状态建议**（与现有 `0=解绑 / 1=绑定` 可扩展）：
  - `pending`：学员申请，待教练同意（若只做「教练邀请」则反向流程）。
  - `active`：已绑定，可指派计划、查看对方授权范围内的数据。
  - `rejected` / `ended`：拒绝或主动解除（保留历史计划只读）。

**权限原则**：所有「为某学员写计划」「看某学员记录」接口，必须先校验 `coach_students.status === active` 且角色匹配。

### 2.2 训练计划：`training_plans`

- 已有字段可支撑：`coach_id`, `student_id`, `title`, `goal`, `start_date`, `end_date`, `description`, `content`(JSON), `status`。
- **区分来源**（与《06-训练中心改造任务书》一致）：
  - **学员自建**：`coach_id == student_id`（或约定 `coach_id=0` 且 `student_id=自己`，二选一全局统一）。
  - **教练指派**：`coach_id != student_id` 且绑定有效。

### 2.3 饮食「计划」：两种落地方式（二选一）

| 方案 | 说明 | 适用 |
|------|------|------|
| **A. 轻量** | 不新建表：教练在学员详情写「饮食建议」文本 + 关联文章 / 食物库链接；学员在饮食中心自行记录。 | 快速上线 |
| **B. 标准** | 新建 `diet_plans`（结构可类比 `training_plans`：`coach_id`, `student_id`, 周期, `content` JSON 每日餐次） | 与训练计划体验对称 |

建议：**一期 A，二期 B**，避免饮食与训练双模块同时大改。

## 三、业务流程（推荐）

### 3.1 学员申请教练

1. 学员在「教练专区」搜索 / 浏览教练（展示：昵称、认证状态、简介、可选评分）。
2. 点击「申请绑定」→ 写入 `coach_students`（`pending`）或独立 `coach_bind_requests`（若需要申请理由、过期时间）。
3. 教练在「学员版块 – 待处理申请」同意 → `active` + `bind_at`；拒绝 → `rejected`。

### 3.2 教练制定训练计划

1. 教练在学员详情选择「新建训练计划」→ `POST` 创建 `training_plans`（`coach_id`=自己，`student_id`=学员）。
2. 学员在「训练中心 – 教练计划」或统一「训练计划」Tab 中查看；仅**查看**，是否允许学员标记「已完成某天」可用 `content` 内状态或单独打卡表扩展。

### 3.3 解绑

- 学员或教练发起「解除绑定」→ `ended`；**已下发的历史计划**仍保留，`student_id` 可见只读，教练端按业务决定是否仍可查看（建议：解绑后教练只读历史、不可再改）。

## 五、接口分层建议（与现有 FastAPI 风格一致）

**学员端（需登录 `user`）**

- `GET /coaches` — 可申请的教练列表（或 `/user/coaches`）
- `GET /coaches/{id}` — 教练公开详情
- `POST /coach-students/apply` —  body: `{ coach_id }`
- `GET /coach-students/me` — 我的绑定状态 / 当前教练
- `DELETE /coach-students/me` — 解除绑定（若开放）
- 训练计划：沿用 `GET /training/plans/coach` 或合并为 `GET /training/plans?source=coach`

**教练端（需登录 `coach`）**

- `GET /coach/students` — 学员列表（含 pending / active 筛选）
- `GET /coach/students/{id}` — 学员详情 + 摘要数据
- `POST /coach/students/{id}/bind` — 同意申请（或 `PUT /coach/bind-requests/{id}`）
- `POST /coach/training/plans` — 为学员创建计划（已有任务书方向）
- `GET /coach/training/plans?student_id=` — 某学员计划列表

**管理员端（需登录 `admin`）**

- `GET /admin/coach-students` — 关系分页 + 筛选
- `GET /admin/coaches/summary`、`GET /admin/students/summary` — 统计看板数据源

具体路径可与现有 `coach_course`、`diet` 路由命名风格统一（`/api` 前缀不变）。

## 六、比「三块页面」略好的产品想法（可选）

1. **单一「关系」真相源**：所有教练–学员能力都依赖 `coach_students`，避免计划接口里散落校验逻辑；中间件或依赖里统一 `require_active_binding(coach_id, student_id)`。
2. **通知占位**：申请通过 / 新计划下发，一期可用站内消息表或前端轮询未读数；二期再接 WebSocket / 推送。
3. **学员隐私档位**：学员可设置「教练可见训练记录范围」（如仅最近 30 天），在 Schema 与接口层过滤。
4. **计划与记录联动**：训练计划中的动作可链接 `actions` 表 ID，学员记训练记录时一键从计划带入 `extra`。
5. **管理员侧重「纠纷与合规」**：导出绑定与计划变更记录，比单纯「看数据」更有运维价值。
