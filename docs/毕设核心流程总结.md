# 毕设核心流程总结

本项目为健身教练平台，涉及三类角色：**用户（学员）**、**教练**、**管理员**，围绕内容投稿审核、课程预约、教练绑定与计划指派四条主线展开。

---

## 一、核心流程概览

### 流程 1：内容投稿与审核（动作库 / 饮食文章 / 食物库）

**参与角色**：教练 → 管理员 → 用户

**流程描述**：

教练在平台上投稿内容（动作、饮食文章、食物），内容初始状态为"待审核"。管理员在后台对投稿进行审核，审核通过后内容上线，用户端可正常浏览；审核驳回时需填写驳回原因，教练可修改后重新提交；管理员也可对已上线内容执行下架操作。

**状态流转**：

```
待审核(0) → 已通过/已上线(1)
待审核(0) → 已驳回(2)  → [教练修改] → 待审核(0)
已上线(1) → 已下架(3)
```

### 流程 2：教练申请审核（用户升级为教练）

**参与角色**：用户 → 管理员

**流程描述**：

普通用户填写申请说明，提交成为教练的申请。管理员在后台审核申请列表，通过后该用户的角色（role）立即变更为 coach，无需重新注册；驳回时需填写拒绝原因，用户可查看审核状态。

**状态流转**：

```
用户提交申请 → pending
pending → approved（role 变为 coach）
pending → rejected（附驳回原因）
```

> 请画一张流程图，描述健身平台的教练申请审核流程，参与角色为普通用户和管理员。流程如下：用户在个人中心填写申请成为教练的说明并提交，申请状态为"待审核"；管理员在后台查看申请列表，可选择通过或拒绝；若通过，系统将该用户的角色字段从 user 修改为 coach，用户立即获得教练权限；若拒绝，管理员填写拒绝原因，用户可在申请状态页查看结果。请用泳道图区分用户和管理员两个角色，状态节点用圆角矩形，判断节点用菱形。

---

### 流程 3：课程发布、预约与审批

**参与角色**：教练 → 管理员 → 用户（学员）

**流程描述**：

教练创建课程并提交，管理员审核通过后课程进入"招募中"状态，用户可浏览并发起预约申请；教练审批预约，通过后已报名人数加一，达到上限后课程自动变为"满员"；课程结束后状态变为"已结束"；管理员可随时下架已上线课程。

**状态流转（课程）**：

```
待审核(0) → 招募中(1) → 满员(2) → 已结束(3)
待审核(0) → 已驳回(4)
招募中(1) / 满员(2) → 已下架(5)
```

**状态流转（预约）**：

```
待审核(1) → 已通过(2)
待审核(1) → 已拒绝(3)
待审核(1) / 已通过(2) → 已取消(0)
```

---

### 流程 4：教练-学员绑定与计划指派

**参与角色**：用户（学员）→ 教练

**流程描述**：

学员在教练列表中选择教练并发起绑定申请，教练在学员管理页审批；绑定成功后，教练可为该学员创建训练计划或饮食计划，学员在训练中心查看教练指派的计划；教练可终止或完成计划，学员也可主动解绑。

**状态流转（绑定关系）**：

```
pending → active（教练同意）
pending → rejected（教练拒绝）
active  → ended（学员解绑）
```

**状态流转（训练/饮食计划）**：

```
创建 → active(1)
active(1) → terminated(0)（教练终止）
active(1) → completed(2)（教练标记完成）
```

## 二、流程关系总览

```
用户注册登录
    │
    ├─→ 申请成为教练 ──→ 管理员审核 ──→ 角色升级为教练
    │
    ├─→ [作为学员] 绑定教练 ──→ 教练审批 ──→ 教练指派训练/饮食计划
    │
    ├─→ [作为学员] 浏览已上线内容（动作/文章/食物/课程）
    │
    └─→ [作为学员] 预约课程 ──→ 教练审批预约

教练登录
    │
    ├─→ 投稿内容（动作/文章/食物）──→ 管理员审核 ──→ 上线/驳回
    │
    ├─→ 发布课程 ──→ 管理员审核 ──→ 招募中 ──→ 审批学员预约
    │
    └─→ 管理绑定学员 ──→ 指派训练/饮食计划

管理员登录
    │
    ├─→ 审核教练申请
    ├─→ 审核内容投稿（动作/文章/食物）
    └─→ 审核课程 / 下架内容
```
